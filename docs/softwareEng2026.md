ソフトウェアが誕生し、開発・運用され、廃止されていくような一連の流れをソフトウェアライフサイクルという。

ソフトウェアを開発するにあたって、まず最初に取り組むものが要件定義である。要件定義書には、システムの目的・概要・機能・構成、目標性能、注意事項、開発スケジュール、納品物などといった項目が含まれており、実現する機能と実現しない機能を明確にすることが必要である。

要件定義は、プロジェクト関係者のバイブルや社内外への宣言、外部委託資料といった役割がある。

要件定義書を製作した後、それをコーディングができるレベルの仕様書や設計図、WBSなどに落とし込む。

制作を行う際は、設計書に沿ってシステム開発を行っていく。

外部委託の際は、ソースコードを納品物に含めるかによって、メンテナンスを自社か外部が行うのかが変わる。また、外部委託をする際は、賃金の安い海外企業に委託するオフショア開発となることもある。その際は、文化や言語、品質への意識といった違いを考慮する必要がある。

開発を進めていくと必ずバグが発生するため、テストとデバッグを行いバグを早期発見、対処をすることが必要である。

開発が終わった後は、運用・保守をする必要があり、開発者以外でも運用可能なように、文書などを残しておく必要がある。

また、海外の人がかかわることもあるため、英語で書くのが好ましい。

プロジェクトは有期性と独自性を持つ。また、勉強などの目標に向けて順に取り組んでいくフォアキャスティングとは違い、プロジェクトは目標から計画を立て、それに取り組むバックキャスティングである。

ソフトウェアを評価する際は、品質（Quality）、コスト（Cost）、納期（Delivery）の三つを用いる。システムによって優先させるものを変える必要がある。

ソフトウェアの評価指標としては、ステップ数、オブジェクト容量、バグ発生率、ファンクションポイント法などがある。

開発プロセスは、様々な方法があり、ウォーターフォール型やスパイラルモデル、反復型、アジャイルといった方法がある。

また、アジャイルの手法として、スクラム、カンバン、XPなどがある。

開発手法を選ぶ際は、人・重要度・変化の度合い・文化・規模の5つの視点で判断することが必要であり、ハイブリッド型にするといった方法もある。

WBSとはプロジェクト目標を達成し、必要な要素成果物を生成するために、プロジェクトチームが実行する作業を、要素成果物を主体に階層的に要素分解したものであり、これによってプロジェクトの見通しが立てやすくなる。

WBSは作業を細かく分割し、それをならべることで作ることができる。

バージョン管理とは、ファイルを「誰が」「いつ」「どのように」変更したかを管理することであり、これらの情報が管理されていれば、多数のメンバーとも共同でコード開発ができる。

バージョン管理には、Subversionなどの集中管理型と、gitなどの分散管理型がある。集中管理型はリモートレポジトリのみがバージョン管理を行うため、同時編集をするとコンフリクトが発生しやすく、同期に時間がかかる。一方、分散管理型はローカルでもバージョン管理を行うため、リモートリポジトリへのアクセス頻度が低く、障害に頑健である。

gitは分散管理型のバージョン管理システムであり、元々はLinus Torvaldsが2005年にオープンソースソフトウェア管理のために開発したソフトウェアである。変更履歴が残る、変更した箇所に戻ることができる、他人と共同編集できるといった特徴がある。

コミットとは、ファイルの作成・変更・削除の記録であり、「いつ・誰が・なにを・どのように」変更したかを残すものである。コミットにはメッセージを残し、「どのように」変更したかを記載する。対象ファイルは一つでも複数でもよく、コミットの単位はユーザーが自由に決定する。

レポジトリとは、gitが管理するプロジェクトのフォルダのことであり、個々のプロジェクト実行環境であるローカルレポジトリと、GithubやGitlabなどの共有の管理場所であるリモートレポジトリがある。

ローカルレポジトリでの流れは、まずワークツリーでファイルを変更して保存し、git addによってコミットしたいファイルをステージングエリアに登録し、git commitによってGitディレクトリに変更を登録するという順になる。ワークツリーのファイルには、gitで管理されていないuntracked、gitで管理されているが変更されていないunmodified、gitで管理されていて変更されたmodifiedという状態がある。ステージングエリアに登録されたファイルはstagedという状態になる。コミットは原則として変更・削除ができず、コミットの変更・削除も記録として残る。また、コミットされたファイルはunmodifiedの状態になる。

リモートレポジトリとのやり取りでは、ローカルレポジトリでcommitされた変更をgit pushによってリモートレポジトリへ反映し、リモートレポジトリにある内容をgit pullによってローカルレポジトリに反映する。なお、リモートレポジトリはワークツリーをもたない。

ブランチを用いると作業を枝分かれさせることができ、共同作業や並行作業が可能になる。ブランチの運用方法には、mainのほかにreleaseやdevelop、featureといった複数のブランチを使い分けるgitフローと、mainとfeatureブランチのみで運用するシンプルなgithubフローがある。

gitの設定は、git config --global user.name や git config --global user.email といったコマンドによって、ホスト（OS）全体のグローバルセッティングとして行う。また、gitに管理してほしくないファイルは.gitignoreに指定する。

Githubはリモートレポジトリのホスティングサービスの一つであり、オープンソースソフトウェアの主要なポータルサイトである。他ユーザーのリモートレポジトリはcloneやforkによって自分の環境に取り込むことができ、templateとして公開されたレポジトリはいくつでもforkできる。

また、Githubには、レポジトリに関する問題・課題・バグ・機能追加や質問などを挙げるためのissues（Redmineでいうticket）や、issuesなどをテーブルやkanban形式で管理するprojectsといった機能があり、これらを活用してソフトウェア・エンジニアリングを行う。演習環境としては、ブラウザ上でVScode環境を構築できるCodespacesを利用する。

マークダウンは簡単に文章を記述するための記法であり、Githubなど多くのアプリケーションで採用されている。拡張子は.mdであり、pandocを使うとtexや.pdf、.docx、.htmlなどに変換できる。
